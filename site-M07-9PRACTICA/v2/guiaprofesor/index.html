    Aquí tens una guia de seguiment per al professorat, pensada per facilitar el control del progrés, l’avaluació formativa i l’orientació didàctica del projecte. Està organitzada per fases i recull tots els conceptes que es treballen:

    📘 Guia de seguiment del professorat#

    Projecte: Sistema de Comunicació Col·laborativa per a Centres de Formació

    🎯 Objectius generals#

    • Aplicar coneixements de desenvolupament web full-stack.
    • Treballar la comunicació entre client i servidor en temps real.
    • Fomentar el treball cooperatiu i la presa de decisions en parella.
    • Introduir funcionalitats col·laboratives com l’edició simultània.

    🔄 Fases i conceptes clau#

    Fase Activitat / Lliurament Conceptes que es treballen Indicadors a observar
    1. Registre i autenticació Formulari de registre i login funcional Validació de formularis, Express, MongoDB, sessions / JWT Connexió base de dades, control d'errors, seguretat
    2. Creació de sales de xat Les parelles tenen una sala assignada automàticament Relacions entre usuaris, identificació de sales, sockets per sales Ús correcte de socket.join, disseny lògic del backend
    3. Xat en temps real Missatges bidireccionals funcionals amb WebSocket socket.emit i socket.on, esdeveniments, backend reactiu Flux de dades correcte, sincronització entre usuaris
    4. Compartició d’arxius Possibilitat de pujar i descarregar fitxers Uploads amb Express, enllaços de descàrrega, seguretat de rutes Validació de tipus de fitxer, persistència mínima
    5. Document col·laboratiu Editor integrat dins la sala amb canvis simultanis socket.io, sincronització d’estat, estructures de dades compartides Canvis sincronitzats, persistència opcional, identificació de la sala
    6. Exportació d’historial Historial de xat i document descarregable en .txt o .json Generació de fitxers, manipulació de cadenes, control de flux Format correcte, descarregues funcionals, contingut vàlid
    7. Frontend responsive Aplicació funcional i agradable visualment React o Vue, CSS modular o framework, ús d’estat Interfície neta, usabilitat bàsica, accés clar a funcionalitats
    8. Treball en parella Seguiment dels rols i la cooperació Organització, comunicació, presa de decisions Interacció entre alumnes, rotació de rols, implicació mútua

    📝 Instruments d’avaluació#

    • ✅ Fitxa de seguiment de la parella (lliuren al final o cada setmana)
    • 👩‍🏫 Observació directa durant el desenvolupament
    • 📁 Revisió del codi (frontend + backend)
    • 💬 Feedback verbal periòdic
    • 🗣️ Presentació final amb demostració de funcionalitats

    🔍 Recomanacions didàctiques#

    • Fer revisions de codi setmanals per detectar males pràctiques.
    • Fomentar l’ús del document col·laboratiu per documentar idees, errors i dubtes.
    • Prioritzar l’enteniment dels fluxos de dades sobre l’estètica.
    • Donar exemples concrets d'esdeveniments socket.io i gestió d’estat compartit.

    ⚠️ Problemes tècnics habituals i solucions#

    Problema Descripció Com solucionar-ho
    Desconnexions WebSocket Els sockets poden tallar-se per latència o inactivitat Comprovar socket.connected, implementar reconexió automàtica amb socket.io
    Missatges repetits L'usuari rep els seus propis missatges duplicats Usar socket.broadcast.emit en lloc de io.emit per evitar autoenviament
    Conflictes en l'edició col·laborativa Dos usuaris editen alhora i un sobreescriu a l’altre Implementar un sistema de bloqueig o controls de versions bàsics (ex: debounce, últim canvi guanya)
    Identificació incorrecta de sales Missatges o documents arriben a sales equivocades Validar correctament roomId, usar socket.join(roomId) i evitar emissions globals
    Fitxers no vàlids o perillosos Pujada de .exe, .js o fitxers massa grans Validar l’extensió i el pes al backend amb multer o equivalent
    Perdre l’estat del document El document col·laboratiu es reinicia si el servidor es reinicia Desar l’estat a la base de dades o a fitxers regularment amb un setInterval o en cada canvi
    Duplicació d’usuaris a parelles Dues persones poden ser assignades a la mateixa parella Comprovar si l’usuari ja té parella abans d’assignar-lo, protegir amb transaccions si cal
    Desincronització entre clients Cada client veu un estat diferent del document o xat Enviar l’estat complet a cada nova connexió i sincronitzar cada canvi
    Frontend no mostra nous missatges El socket.on(...) no s’activa o el component no es renderitza Comprovar que el socket s’instancia fora del component, i revisar l’ús de useEffect correctament
    Errors CORS El navegador bloqueja les peticions al servidor Configurar cors al backend (cors({ origin: \"http://localhost:3000\" }))
    Usuari desconegut a l'inici de sessió Problemes amb identificació o validació Validar dades, usar bcrypt per les contrasenyes i retornar errors clars al frontend
    Accés a sales no autoritzades Un alumne entra en una sala que no li correspon Comprovar al servidor si el roomId coincideix amb la parella de l’usuari abans de permetre l’accés
    Persistència parcial o inconsistent L’usuari veu dades que no coincideixen amb la base de dades Assegurar que cada acció socket té la seva versió equivalent POST/PUT per guardar les dades també via API
