    Ejercicios prácticos IoT Console#

    1. Añadir múltiples dispositivos#

    Objetivo: Simular al menos 3 dispositivos distintos (sensor_A, sensor_B, sensor_C) y añadir lecturas manuales o con el simulador para cada uno.

    Indicaciones:

    • Usar el menú para insertar datos manualmente.
    • Ejecutar simulate.py con distintos device_id.

    2. Listar lecturas por fecha concreta#

    Objetivo: Mostrar todas las lecturas del sensor_A entre dos fechas dadas (por ejemplo: del 1 de mayo al 3 de mayo).

    Indicaciones:

    • Usar el menú y filtrar por device_id y rango de tiempo.
    • Explora qué ocurre si introduces solo fecha inicio o solo fecha fin.

    3. Activar alarma si temperatura supera umbral#

    Objetivo: Implementar una función que muestre un mensaje de “¡ALERTA!” si alguna lectura supera los 28 °C.

    Indicaciones:

    • Recorrer lecturas recientes con get_readings().
    • Filtrar con if reading["data"]["temperature"] > 28:.

    4. Listar lecturas por hora, sin importar el dispositivo#

    Objetivo: Obtener todas las lecturas que se realizaron entre las 14:00 y 15:00 del 2 de mayo, sin filtrar por device_id.

    Indicaciones:

    • Solo filtrar por timestamp.
    • Usa datetime.combine() para definir la hora exacta.

    5. Comparar dos sensores en un mismo intervalo#

    Objetivo: Mostrar en paralelo las lecturas de sensor_A y sensor_B entre el 1 y el 2 de mayo.

    Indicaciones:

    • Llamar dos veces a get_readings() con mismo rango de tiempo.
    • Mostrar resultados alternando sensores o en tablas paralelas.

    6. Calcular promedio diario de cada sensor#

    Objetivo: Calcular la temperatura media diaria de cada sensor a lo largo de 3 días.

    Indicaciones:

    • Agrupar lecturas por fecha (timestamp.date()).
    • Calcular la media por día y por sensor.

    7. Obtener última lectura de cada sensor#

    Objetivo: Mostrar la lectura más reciente de todos los sensores disponibles.

    Indicaciones:

    • Obtener lista de device_id distintos (distinct("device_id")).
    • Para cada uno, buscar find_one() ordenado por timestamp descendente.

    8. Generar listado con diferencias de temperatura hora a hora#

    Objetivo: Para un sensor específico, calcular cuánto varió la temperatura entre una hora y la siguiente.

    Indicaciones:

    • Agrupar lecturas por hora.
    • Calcular diferencia entre cada lectura consecutiva.

    9. Simular y guardar datos cada X segundos (avanzado)#

    Objetivo: Modificar simulate.py para guardar datos de dos sensores alternando cada 5 segundos.

    Indicaciones:

    • Alternar device_id dentro del bucle con un contador.
    • Usar un try-except para controlar interrupciones.

    10. Exportar lecturas combinadas a JSON#

    Objetivo: Exportar a un archivo lecturas.json todas las lecturas de sensor_A y sensor_B en el mismo archivo, en orden cronológico.

    Indicaciones:

    • Usar get_readings() con ambos sensores.
    • Unir listas, ordenarlas por timestamp y escribir con json.dump().

    Anexos#

    Documentación: Clase Reading#

    📄 Descripción General#

    La clase Reading representa una lectura de sensor IoT destinada a ser almacenada en una base de datos MongoDB, especialmente dentro de una colección time-series. Su estructura está diseñada para garantizar compatibilidad con los requisitos de MongoDB en cuanto a almacenamiento temporal eficiente.

    📊 Propósito#

    Facilitar la creación de documentos que contienen datos de sensores con:

    • Marca temporal (timestamp)
    • Metadatos del dispositivo
    • Datos registrados (ej. temperatura o índice de humedad)
    • Identificador directo del dispositivo

    Estos documentos son luego insertados en MongoDB mediante funciones como insert_one().

    🔢 Definición de la Clase#

    from datetime import datetime

class Reading:
    def __init__(self, device_id, value, variable_type="temperature", timestamp=None):
        self.timestamp = timestamp or datetime.utcnow()
        self.metadata = {
            "device_id": device_id,
            "variable_type": variable_type
        }
        self.data = { variable_type: value }
        self.device_id = device_id

    🔍 Atributos#

    Atributo Tipo Descripción
    timestamp datetime Fecha y hora de la lectura. Si no se proporciona, se usa la hora actual en UTC.
    metadata dict Diccionario con información contextual. Incluye device_id y el tipo de variable medida.
    data dict Datos reales de la lectura, como temperature o humidity_index.
    device_id str Identificador del sensor que tomó la lectura.

    🔄 Ejemplo de Uso#

    from schemas import Reading
from models import add_reading

# Crear una lectura de temperatura
lectura_temp = Reading("sensor_A", 23.5, variable_type="temperature")

# Crear una lectura de índice de humedad
lectura_humedad = Reading("sensor_B", 60.2, variable_type="humidity_index")

# Insertar en la base de datos
add_reading(lectura_temp)
add_reading(lectura_humedad)

    Esto genera un documento como:

    {
  "timestamp": "2025-05-12T15:00:00Z",
  "metadata": {
    "device_id": "sensor_B",
    "variable_type": "humidity_index"
  },
  "data": {
    "humidity_index": 60.2
  },
  "device_id": "sensor_B"
}

    🚀 Integración con MongoDB Time-Series#

    MongoDB utiliza colecciones especiales para datos temporales. Esta clase respeta la estructura recomendada para ellas:

    • timestamp: campo obligatorio como referencia temporal.
    • metadata: campo opcional recomendado para agrupar lecturas por dispositivo.
    • data: contenedor del valor medido.

    Esta estructura optimiza:

    • Almacenamiento en "buckets"
    • Compresión automática
    • Consultas por tiempo o dispositivo

    🔧 Buenas prácticas#

    • Usar datetime.utcnow() para mantener coherencia horaria.
    • Separar metadatos de los datos medidos.
    • Evitar campos adicionales innecesarios que compliquen las consultas.

    ✅ Conclusión#

    La clase Reading proporciona una interfaz sencilla y estructurada para capturar lecturas de sensores y almacenarlas en MongoDB de forma eficiente y escalable. Es fundamental en proyectos IoT donde la organización temporal y la flexibilidad de los datos son claves. Ahora además permite registrar distintas variables como temperatura o índice de humedad, haciendo el sistema más versátil.
